# 읽기 좋은 기술문서 작법 (readable-docs)

## 언제 쓰나

코드를 근거로 기술/API 문서를 생성하거나 다듬을 때. AI 기본 출력은 정확하지만 장황하고 요약이 안 되어 사람이 다시 고쳐야 한다. 이 스킬은 그 재작업을 없앤다.

## 작성 원칙 (순서대로 적용)

1. **TL;DR 먼저** — 문서 맨 위 3~5줄로 "무엇을 / 누가 / 언제 쓰나"를 요약한다. 상세는 그 다음.
2. **점진적 공개** — 개요 → 핵심 개념 → 사용 예시 → 상세 레퍼런스 순. 독자가 필요한 깊이에서 멈출 수 있게 한다.
3. **나열 대신 구조** — 파라미터·옵션·반환값은 표로. 비슷한 항목 반복은 표/리스트로 압축한다.
4. **예시 우선** — 추상 설명보다 짧고 실행 가능한 예시 1개를 먼저 보여준다.
5. **무자비한 압축** — 자명한 문장, 중복, "보일러플레이트 산문"을 삭제한다. 한 문장에 한 가지만.
6. **독자 검증** — "이 모듈을 처음 본 팀원이 이 문서만 보고 5분 안에 쓸 수 있나?"를 자문한다.
7. **이모지 금지** — 문서에 이모지·이모티콘을 쓰지 않는다. 상태·강조는 텍스트로 표현한다(예: "완료", "주의", **굵게**).

## Before / After

**Before (장황한 기계 덤프):**

    이 함수는 사용자 인증을 처리하는 함수입니다. 이 함수는 username과 password라는
    두 개의 파라미터를 받습니다. username 파라미터는 문자열 타입이며 사용자의 아이디를
    나타냅니다. password 파라미터도 문자열 타입이며 사용자의 비밀번호를 나타냅니다.
    이 함수는 인증에 성공하면 토큰을 반환하고 실패하면 예외를 발생시킵니다.

**After (읽기 좋게):**

    ## authenticate(username, password)

    사용자를 인증하고 액세스 토큰을 반환한다. 실패 시 AuthError.

    | 파라미터 | 타입 | 설명 |
    |---|---|---|
    | username | str | 사용자 아이디 |
    | password | str | 비밀번호 |

    예시:  token = authenticate("alice", "pw")

## 마무리 self-check

- [ ] 맨 위 3~5줄 TL;DR이 있는가
- [ ] 파라미터/옵션이 표로 정리됐는가
- [ ] 실행 가능한 예시가 최소 1개 있는가
- [ ] 빈 줄 없는 벽글 문단이 없는가
- [ ] 처음 보는 팀원이 5분 안에 쓸 수 있는가
- [ ] 이모지·이모티콘을 쓰지 않았는가
