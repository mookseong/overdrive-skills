# readable-docs

AI가 만든 기술/API 문서가 정확하지만 **장황하고 핵심 요약이 안 되어** 사람이 매번 다시 다듬는 문제를 푼다. 문서를 자주 쓰는 개발자가 대상이다.

## 무엇이 들어 있나

| 컴포넌트 | 역할 |
|---|---|
| **Skill** `readable-docs` | 읽기 좋은 한국어 기술문서 작법 6원칙을 강제 (품질의 핵심) |
| **Hook** PostToolUse | 문서 저장 시 구조 누락 3종을 잡아 자동 수정 유도 (안전망) |

Hook이 잡는 것(구조 프록시): ①상단 요약(TL;DR) 누락 ②예시/코드블록 누락 ③벽글 문단.
**대상 파일:** `docs/` 하위 `*.md`와 최상위 `README.md`만 검사한다(그 외 마크다운엔 발동 안 함).
**한계:** "읽기 좋음"은 정성적이라 스크립트로 완전 검증 불가. 품질은 스킬이 책임지고 hook은 명백한 누락만 잡는다.

## 설치

```bash
/plugin marketplace add mookseong/overdrive-skills
/plugin install readable-docs@overdrive-skills
```

설치 후 Claude Code 재시작.

## 데모 (회의용)

1. **베이스라인**: 실제 모듈에 "이 API 문서화해줘" → 장황한 출력 확보.
2. **적용**: 동일 요청 → 스킬이 읽기 좋은 문서 생성, hook이 구조 검증.
3. **before/after** 비교 + 정량: 재작성 라운드 0회 / 분량 X→Y자 / 가독성 체크 통과.

## 회의 논의포인트 매핑

- **시간 절감**: 재작성 라운드 횟수↓, 분량↓, 팀원 이해 시간↓
- **AI 실패→우회**: 장황·요약 안 됨 → 스킬로 작법 강제 + hook로 구조 게이트
- **팀 표준**: 스킬 = 팀 문서 스타일 가이드, hook = 자동 가드레일

## 한계 / 알려진 제약

- Hook은 macOS/Linux에서 `python3` 직접 호출. Windows는 `python`이 PATH에 있어야 함.
- Hook은 구조 프록시 3종만 검사한다(가독성 자체를 보장하지 않음).
